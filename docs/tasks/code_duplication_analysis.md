# Code Duplication Analysis (Plan & Analyze — No Code Changes)

## Overview

**Goal:** Plan and systematically analyze code duplication across the Display Commander addon so that future refactors can be prioritized and done safely. This task is **analysis and documentation only**: no code is modified, so it can run in parallel with other development.

**Scope:** `src/addons/display_commander/` and addon-specific code under `src/`. External submodules (reshade, nvidia-dlss, etc.) are out of scope.

**Constraints:**
- **Read-only:** Do not change any source files; only add/update this task doc and any analysis artifacts (e.g. lists, tables).
- **Parallel-safe:** Other people may be working on the codebase at the same time; analysis should not depend on a frozen codebase.
- **Multi-phase, high effort:** Multiple sub-tasks; expect significant time. Do planning and analysis first; refactoring is a later, separate effort.

---

## Definitions

| Term | Definition |
|------|------------|
| **Exact duplication** | Two or more identical or near-identical code blocks (copy-paste). |
| **Structural duplication** | Same pattern repeated with small variations (e.g. same hook install pattern in many files, same settings init pattern per tab). |
| **Generated duplication** | Repetitive code produced by scripts (e.g. proxy stubs, OpenGL forwarders). May be intentional; document and mark "by design." |
| **API / helper underuse** | A shared helper or API exists (e.g. `CreateAndEnableHook`) but some call sites still implement the same logic inline. |
| **Template-worthy pattern** | Repeated logic that could be factored into a template, macro, or shared function without changing behavior. |

---

## Approach

1. **Plan (this doc)**
   - Define duplication types and scope.
   - List analysis sub-tasks and order.
   - Define deliverables (lists, tables, recommendations only — no patches).

2. **Analyze by category**
   - Hooks: MinHook install/uninstall patterns, error handling, logging.
   - Settings/UI: Tab settings (all_settings_, Load/Save, GetAllSettings), checkbox/slider patterns.
   - Locking: SRWLOCK usage (already centralized in `utils::SRWLock*`; note any remaining ad-hoc lock patterns).
   - Proxies / dynamic loading: GetProcAddress/LoadLibrary patterns (e.g. proxy_dll, dbghelp_loader, version.dll).
   - Logging and error messages: Repeated format strings or log patterns.

3. **Document findings**
   - For each finding: location (file, optional line range), type (exact/structural/generated/underuse/template-worthy), severity (low/medium/high), and a short recommendation (e.g. "Consider using CreateAndEnableHook" or "Keep as-is, generated by script").
   - No code changes in this phase.

4. **Prioritize (later)**
   - After analysis, maintain a short "recommended refactors" list with priority; actual refactors are separate PRs.
