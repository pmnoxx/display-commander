# Vulkan-1 proxy audit plan

## Goal

Check the Display Commander `vulkan-1` proxy against the **official** Vulkan API (Khronos headers). Do **not** use the existing proxy or exports as the source of truth; use the spec derived from official documentation only.

## Spec (official API list)

- **File:** `scripts/specs/vulkan-1_official_api_list.txt`
- **Source:** Khronos Vulkan-Headers — `vulkan_core.h` and `vulkan_win32.h` (PFN_vk* typedefs, excluding callback typedefs).
- **Generated by:** `python scripts/extract_vulkan_api_from_headers.py --win32`
- **Count:** 738 API names (one per line, sorted).

Regenerating the list:

```bash
python scripts/extract_vulkan_api_from_headers.py --win32
```

## Audit process: one API at a time

1. **For each API** in `vulkan-1_official_api_list.txt` (in order, or by category):
   - **Present in proxy?**  
     Check that the proxy exports and implements the symbol (e.g. in `proxy_dll/vulkan-1_proxy.cpp` and `proxy_dll/exports.def`).
   - **Signature correct?**  
     Compare proxy declaration/forwarding against the official header (same parameter types, return type, calling convention).
   - **Behavior correct?**  
     Proxy should forward to the real loader/driver; no missing null checks, wrong order of calls, or incorrect error handling where the spec defines it.

2. **Track results** in a simple way, e.g.:
   - OK — present and matches spec
   - Missing — in spec but not in proxy
   - Wrong signature — parameter/return type or convention mismatch
   - Wrong behavior — forwarding or semantics don’t match spec

3. **Optional grouping** (for batching):
   - Instance/device creation and enumeration
   - Command buffer / queue submission
   - Memory, buffer, image
   - Pipeline, descriptor, render pass
   - Swapchain / presentation (KHR, EXT, etc.)
   - Extensions (KHR, EXT, NV, AMD, etc.)

## Proxy files to check

- `src/addons/display_commander/proxy_dll/vulkan-1_proxy.cpp` — implementations and forwarding.
- `src/addons/display_commander/proxy_dll/exports.def` — exported symbols for vulkan-1.

## Official references

- Vulkan 1.4 Specification: https://registry.khronos.org/vulkan/specs/latest/html/
- Vulkan API Registry: https://registry.khronos.org/vulkan/specs/latest/registry.html
- Vulkan-Headers (vk.xml, vulkan_core.h): https://github.com/KhronosGroup/Vulkan-Headers

## Automated comparison: proxy vs doc (headers)

A script compares the proxy to the **official Vulkan headers** (not the spec list) and reports mismatches:

```bash
python scripts/compare_vulkan_proxy_to_doc.py --win32 -o docs/vulkan_proxy_vs_doc_report.txt
```

**Report:** `docs/vulkan_proxy_vs_doc_report.txt`

- **Missing in proxy:** APIs declared in vulkan_core.h / vulkan_win32.h but not in vulkan-1_proxy.cpp.
- **In proxy not in doc:** Symbols in the proxy with no matching PFN in the headers (e.g. vendor/legacy).
- **Parameter count mismatch:** Doc says N parameters, proxy has M; fix the Wine spec or generator.
- **Return type mismatch:** Doc says `void` / `VkResult` / function pointer; proxy uses `int` or `void` incorrectly (e.g. `void` functions returning `int`, or `GetInstanceProcAddr`/`GetDeviceProcAddr` returning `int` instead of a pointer).

Use the report to fix the generator/spec and then re-run to verify.

## Summary

- **Spec:** `scripts/specs/vulkan-1_official_api_list.txt` (from official headers only).
- **Process:** For each API in the spec, verify presence, signature, and behavior in the proxy.
- **Comparison vs doc:** Run `compare_vulkan_proxy_to_doc.py` to get mismatches against the headers (see report above).
- **Do not** derive the spec from the existing vulkan-1 proxy or exports.
